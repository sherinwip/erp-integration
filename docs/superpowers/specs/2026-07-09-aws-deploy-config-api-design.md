# erp-config-api AWS Deployment — Design

## Goal

Deploy `erp-config-api` (FastAPI CRUD service) to AWS, backed by a managed
Postgres (RDS), with the `database/liquibase` changelog applied against
that RDS instance. All infra defined as Terraform, committed on a new
branch. No changes to existing local dev workflow (docker-compose,
`.env`-based local Postgres) — this is additive.

## Scope

In scope:
- Terraform-defined AWS infra: RDS Postgres, ECR repo, ECS Fargate
  cluster/service/task, ALB, security groups, IAM roles, Secrets Manager
  secret for DB credentials.
- A production-style `Dockerfile` for `erp-config-api` (none exists today).
- A one-off ECS task definition that runs the existing `database/liquibase`
  Docker image against RDS to apply the changelog.
- New git branch for all of this work.

Out of scope:
- transformation-svc deployment (separate future effort).
- erp-config-portal deployment.
- CI/CD pipeline (deploys are manual `terraform apply` / `aws ecs run-task`
  for now).
- HTTPS/ACM cert on the ALB (HTTP only for this pass — flagged as a known
  gap, not a blocker).
- Multi-AZ / prod-hardened RDS (single-AZ, small instance — this is a first
  deployment, not a HA prod setup).

## Architecture

- **Region:** us-east-1
- **Network:** default VPC (existing, not Terraform-managed), using its
  existing public subnets across 2+ AZs.
- **Database:** RDS Postgres 16, `db.t4g.micro`, single-AZ, 20GB gp3,
  not publicly accessible. Matches the local `postgres:16` image
  (`IAC/docker-compose.yml`) and `erp_integration` DB name/schema.
  Master password auto-generated by Terraform (`random_password`), stored
  in AWS Secrets Manager.
- **Container registry:** new ECR repo `erp-config-api`. Image built
  locally from a new `erp-config-api/Dockerfile` (python:3.12-slim,
  installs `requirements.txt`, runs
  `uvicorn app.main:app --host 0.0.0.0 --port 8010`), pushed via
  `docker build` + `docker push` (not part of `terraform apply` itself —
  Terraform creates the repo, push is a documented manual/CLI step run
  once before first `apply` of the ECS service, and again on each image
  update).
- **Compute:** ECS Fargate cluster, one service running `erp-config-api`,
  desired count 1, task reads `DB_HOST/DB_PORT/DB_NAME/DB_USER/DB_PASSWORD`
  from the Secrets Manager secret via the task execution role
  (`secrets` block in the container definition, not env vars baked into
  the image).
- **Load balancing:** internet-facing ALB, HTTP:80 → target group →
  Fargate service container port 8010. No HTTPS in this pass.
- **Security groups:**
  - ALB SG: ingress 80/tcp from 0.0.0.0/0
  - ECS SG: ingress 8010/tcp from ALB SG only
  - RDS SG: ingress 5432/tcp from ECS SG only (and from the one-off
    liquibase task's SG/role)
- **Liquibase apply:** a separate ECS Fargate task definition (not a
  service — run-once), using the existing `database/liquibase` Dockerfile
  and changelog, pointed at the RDS endpoint via the same Secrets Manager
  secret. Terraform creates the task definition; applying the changelog is
  a manual `aws ecs run-task` invocation run once after RDS is up and
  before the config-api service is expected to serve real traffic (and
  again whenever the changelog changes).

## Data flow

1. `terraform apply` provisions VPC networking references, RDS, Secrets
   Manager secret, ECR repo, IAM roles, ECS cluster, ALB, target group,
   security groups, and both ECS task definitions (config-api service +
   liquibase one-off).
2. Operator builds and pushes the `erp-config-api` image to the new ECR
   repo.
3. Operator runs the liquibase one-off task via `aws ecs run-task` to
   migrate the RDS schema.
4. ECS service starts config-api tasks, pulling from ECR, reading DB
   creds from Secrets Manager, registering with the ALB target group.
5. ALB DNS name is the entry point for testing (`curl` / Postman against
   `http://<alb-dns>/api/v1/clients` etc., mirroring today's
   `localhost:8010` smoke tests).

## Error handling / operational notes

- If the liquibase task fails, the config-api service will still start
  but fail at first query (missing schema) — health check (`/docs` or a
  future `/health`) won't catch this since it doesn't touch the DB;
  operator must check ECS task logs / run liquibase task synchronously
  and confirm exit code 0 before relying on the service.
- Secrets Manager secret is the single source of truth for DB credentials
  post-deploy; no `.env` file is created or used in AWS.
- All resources tagged (e.g. `Project=erp-integration`,
  `ManagedBy=terraform`) for later cleanup/cost tracking.

## Testing

- After `terraform apply` + image push + liquibase run: `curl` the ALB
  DNS `/docs` (expect 200) and `/api/v1/clients` (expect data, mirroring
  the local smoke test already done for `localhost:8010`).
- No automated infra tests in this pass (out of scope) — manual
  verification only, consistent with how local services were smoke-tested
  earlier in this session.

## Git workflow

- New branch: `feature/aws-deploy-config-api`, created from `main`.
- Terraform code lives under `IAC/terraform/` (currently empty).
- Dockerfile added at `erp-config-api/Dockerfile`.
- Changes committed to the branch; **not pushed to remote** unless
  explicitly requested.
